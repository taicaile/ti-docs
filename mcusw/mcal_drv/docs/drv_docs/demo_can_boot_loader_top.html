Introduction

This application demonstrates execute-in-place (XIP) feature provided by Octal SPI flash controller on J721E/AM65xx SoC. The application performs following actions:

  1. MCAL Can driver is hosted on MCU 1 0
  2. Transmit a 64 byte CAN message
  3. Boot MPU1_0 (e.g. A53) with a sample application image

This facilitates an early CAN response feature, typically required in an automotive ECU.

Internal Dependencies

This application depends on multiple components and are detailed in sections below

  1. MCAL CAN Driver
  2. MCAL BSW Stubs Stubs at processor_sdk_rtos_am65xx_xx.yy.xx.bb/mcusw_xx.yy.xx.bb/mcal_drv/mcal/Bsw_Stubs are used. Functions CanIf_TxConfirmation () is required.
  3. MCAL Configurations CAN & DIO Configurations at processor_sdk_rtos_am65xx_xx.yy.xx.bb/mcusw_xx.yy.xx.bb/mcuss_demos/mcal_config are used.

Building dependent binaries

We need have the following bianries built from processor_sdk_rtos_am65xx_xx.yy.xx.bb:

  1. sbl_cust_img that loads SYSFW, initializes the OSPI flash in execute-in-place (XIP) mode
  2. sbl_boot_xip_entry that provides the entry point to the binary which executes from OSPI
  3. sbl_boot_test sample binary that runs on MPU1_0

Apart from the above binaries, we need to build the can_boot_app that runs from OSPI flash in XIP mode, and does the following:

  1. CAN loopback / transmit
  2. Boot MPU1_0

Building sbl_cust_img

Go to processor_sdk_rtos_am65xx_xx.yy.xx.bb/pdk_am65xx_$version/packages/ti/build and run the following: 

make BOARD=am65xx_evm SOC=am65xx sbl_cust_img

Building sbl_boot_xip_entry

Go to processor_sdk_rtos_am65xx_xx.yy.xx.bb/pdk_am65xx_$version/packages/ti/build and run the following: 

make BOARD=am65xx_evm SOC=am65xx sbl_boot_xip_entry

Building sbl_boot_test for MPU1_0

Go to processor_sdk_rtos_am65xx_xx.yy.xx.bb/pdk_am65xx_$version/packages/ti/build and run the following: 

make BOARD=am65xx_evm SOC=am65xx CORE=mpu1_0 sbl_boot_test

Building can_boot_app

Go to processor_sdk_rtos_am65xx_xx.yy.xx.bb/mcusw_xx.yy.xx.bb/build and run the following: 

make can_boot_app

Steps to run

Flash the following binaries in OSPI memory at the respective offsets (in hex) as given below:

  1. processor_sdk_rtos_am65xx_xx.yy.xx.bb/pdk_am65xx_$version/packages/ti/boot/sbl/binary/am65xx_evm/cust/bin/sbl_cust_img_mcu1_0_release.tiimage @ 0
  2. processor_sdk_rtos_am65xx_xx.yy.xx.bb/prebuilt-sdcards/am65xx_evm/sd_card_files/sysfw.bin @ 40000
  3. processor_sdk_rtos_am65xx_xx.yy.xx.bb/pdk_am65xx_$version/packages/ti/boot/sbl/example/k3MulticoreApp/binary/am65xx/sbl_baremetal_boot_xip_entry_am65xx_evm_mcu1_0TestApp_release.appimage @ a0000
  4. processor_sdk_rtos_am65xx_xx.yy.xx.bb/mcusw_xx.yy.xx.bb/binary/can_boot_app/bin/am65xx_evm/can_boot_app_mcu1_0_release.xer5f.bin @ e0000
  5. processor_sdk_rtos_am65xx_xx.yy.xx.bb/pdk_am65xx_$version/packages/ti/boot/sbl/example/k3MulticoreApp/binary/am65xx/sbl_baremetal_boot_test_am65xx_evm_mpu1_0TestApp_release.appimage @ 2e0000

Refer Uniflash Programming Guide to learn how to program the binaries at the respective offset in the flash.

Detail Steps to run the demo is provided at User Guide

After bootlaoding of MPU1_0 is completed, MCU1 R5F prints the profiling information from the start of SBL to the point when can_boot_app starts execution.
